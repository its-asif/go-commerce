# GO Commerce Project Description

## 1. Project Overview

GO Commerce is a backend REST API for an electronic commerce platform implemented in Go. It focuses on core commerce workflows:

- User registration and login
- Role-aware access control (user/admin)
- Product and category management
- Shopping cart operations
- Checkout and order history
- API documentation via Swagger
- Runtime performance optimization via Redis caching

The system is designed as a monolithic HTTP service with clear package-level separation for handlers, data access, middleware, models, configuration, and utilities.

## 2. High-Level Architecture

The application follows a layered request flow:

1. HTTP request enters Gorilla Mux router.
2. Route matching determines handler and whether middleware is applied.
3. `AuthMiddleware` parses Bearer JWT and injects user identity/role into request context.
4. `AdminMiddleware` (where used) enforces admin authorization.
5. Handler validates input, optionally checks Redis cache, and reads/writes PostgreSQL through `db` package or direct queries.
6. Handler returns JSON response and invalidates/sets cache keys as needed.

Core architectural properties:

- Single-process API service (`main.go`) with shared global DB and Redis clients (`db.DB`, `db.Rdb`).
- Persistence in PostgreSQL.
- Read optimization with key-based Redis cache for common list/detail resources.
- API docs generated by `swag` and served at `/swagger/index.html`.

## 3. Technology Stack

- Language: Go `1.24.4`
- HTTP Router: `github.com/gorilla/mux`
- SQL Access: `github.com/jmoiron/sqlx`
- PostgreSQL Driver: `github.com/lib/pq`
- Redis Client: `github.com/redis/go-redis/v9`
- JWT: `github.com/golang-jwt/jwt/v5`
- Password Hashing: `golang.org/x/crypto/bcrypt`
- Env Loading: `github.com/joho/godotenv`
- Swagger: `github.com/swaggo/http-swagger` + generated docs in `docs/`
- Development reload: `air` (installed in Dockerfile)

## 4. Runtime Boot Process

Startup sequence in `main.go`:

1. Print welcome message.
2. Load environment variables from `.env` with `config.LoadEnv()`.
3. Connect PostgreSQL via `db.ConnectDB()` using `DB_URL`.
4. Connect Redis via `db.ConnectRedis()` using `REDIS_URL`, `REDIS_PASSWORD`, `REDIS_DB`.
5. Register routes through `routes.GetRoutes(router)`.
6. Mount Swagger UI at `/swagger/`.
7. Read `PORT` and start HTTP server with `http.ListenAndServe`.

If env loading fails, process exits (`log.Fatal`).

## 5. Directory and Component Responsibilities

### `config/`

- `env.go`: `.env` loading and environment retrieval helpers.

### `db/`

- `db.go`: Initializes global PostgreSQL (`DB`) and Redis (`Rdb`) clients.
- `user_db.go`: User retrieval helpers (`GetAllUsers`, `GetSingleUser`).
- `product_db.go`: Product CRUD data-layer helpers including dynamic update query construction.

### `handlers/`

HTTP endpoint implementations:

- `handlers.go`: root ping/home endpoint (`/`).
- `health.go`: health endpoint (`/healthz`) returning `{ "status": "ok" }`.
- `auth_handler.go`: register/login.
- `user_handler.go`: admin user listing and lookup endpoints.
- `products.go`: product CRUD endpoints, with caching.
- `category_handler.go`: category create/list endpoints, with caching.
- `cart_handler.go`: cart list/add/remove endpoints, with caching.
- `order_handler.go`: checkout and order history endpoints, with caching.

### `middleware/`

- `auth_middleware.go`: JWT auth middleware, context key definitions, admin authorization middleware.

### `models/`

Typed request/response/domain structures for users, products, categories, cart items, orders, and Swagger schema metadata.

### `routes/`

- `routes.go`: central route registration, route grouping, and middleware assignment.

### `utils/`

- `hashing.go`: bcrypt hashing/checking.
- `jwt.go`: token generation and parsing with expiration and algorithm checks.
- `redis_cache.go`: JSON-based get/set/delete cache helpers.

### `migrations/`

SQL schema migration files for users, categories, products, cart_items, orders, and order_items.

### `docs/`

Swagger generated files (`docs.go`, `swagger.json`, `swagger.yaml`).

## 6. Data Model (PostgreSQL Schema)

### Users

Table: `users`

- `id SERIAL PRIMARY KEY`
- `name TEXT NOT NULL`
- `email TEXT UNIQUE NOT NULL`
- `password TEXT NOT NULL`
- `role TEXT DEFAULT 'user'`
- `address TEXT`
- `phone TEXT`
- `avatar TEXT`
- `last_login TIMESTAMP`
- `status TEXT DEFAULT 'active'`
- `created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`

### Categories

Table: `categories`

- `id SERIAL PRIMARY KEY`
- `name TEXT NOT NULL`
- `slug TEXT UNIQUE NOT NULL`

### Products

Table: `products`

- `id SERIAL PRIMARY KEY`
- `name TEXT NOT NULL`
- `description TEXT`
- `price NUMERIC(10,2) NOT NULL`
- `image_url TEXT`
- `stock INT NOT NULL`
- `category_id INT REFERENCES categories(id)`
- `created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`

### Cart Items

Table: `cart_items`

- `user_id INT REFERENCES users(id)`
- `product_id INT REFERENCES products(id)`
- `quantity INT NOT NULL`
- `price NUMERIC(10,2) NOT NULL`
- `added_at TIMESTAMP DEFAULT now()`
- `PRIMARY KEY (user_id, product_id)`

This composite key enables UPSERT behavior by `(user_id, product_id)`.

### Orders

Table: `orders`

- `id SERIAL PRIMARY KEY`
- `user_id INT REFERENCES users(id)`
- `total_price NUMERIC(12,2) NOT NULL`
- `status TEXT DEFAULT 'pending'`
- `placed_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP`

### Order Items

Table: `order_items`

- `id SERIAL PRIMARY KEY`
- `order_id INT REFERENCES orders(id)`
- `product_id INT REFERENCES products(id)`
- `quantity INT NOT NULL`
- `price NUMERIC(10,2) NOT NULL`

## 7. API Surface and Access Control

### Public Endpoints

- `GET /` -> basic service status message.
- `GET /healthz` -> health probe JSON.
- `POST /auth/register` -> create user with bcrypt password hash.
- `POST /auth/login` -> issue JWT token.

### Authenticated Endpoints (`/api` with `AuthMiddleware`)

- `GET /api/products`
- `GET /api/products/{id}`
- `PUT /api/products/{id}`
- `GET /api/cart`
- `POST /api/cart`
- `DELETE /api/cart/{product_id}`
- `POST /api/orders/checkout`
- `GET /api/orders`

### Admin Endpoints (`/api` + `AdminMiddleware`)

- `GET /api/users`
- `GET /api/users/id/{id}`
- `GET /api/users/email/{email}`
- `POST /api/products`
- `DELETE /api/products/{id}`
- `POST /api/categories`
- `GET /api/categories`

Notes on authorization behavior:

- JWT claims may include `role`; if present and `admin`, admin middleware allows request immediately.
- If role claim is missing/non-admin, middleware queries `users.role` in DB for final decision.

## 8. Authentication and Security Mechanics

### Registration

- Accepts name/email/password.
- Hashes password with bcrypt default cost.
- Inserts user into `users` table.
- Returns `201` with partial user data (`id`, `created_at` fields populated in response struct).
- Unique email conflict (`pq` code `23505`) returns `400`.

### Login

- Accepts email/password.
- Attempts to read full user row from Redis cache key `user_email_<trimmed_email>`.
- Falls back to DB query on cache miss.
- Compares bcrypt hash.
- Issues JWT with:
  - `uid`
  - `role`
  - `exp` (24h)
  - `iat`

### JWT Parsing/Validation

- Enforces HMAC signing method.
- Uses `JWT_SECRET`.
- Checks token validity and explicit `exp` handling for map claims.
- `AuthMiddleware` requires `Authorization: Bearer <token>` format.

## 9. Caching Strategy (Redis)

Caching uses JSON serialization for complex objects and TTL by data type.

### Cache Keys and TTLs

- User by email: `user_email_<email>` -> 15 minutes
- All products: `all_products` -> 10 minutes
- Single product: `product_<id>` -> 5 minutes
- All categories: `all_categories` -> 30 minutes
- Cart by user: `cart_user_<userId>` -> 5 minutes
- Orders by user: `orders_user_<userId>` -> 10 minutes

### Invalidation Rules

- Product create/delete/update invalidates `all_products`; update/delete also invalidate `product_<id>`.
- Category create invalidates `all_categories`.
- Cart add/remove invalidates `cart_user_<userId>`.
- Checkout invalidates both `cart_user_<userId>` and `orders_user_<userId>`.

This design improves read performance and keeps cache coherence by explicit mutation-side invalidation.

## 10. Core Business Flows

### Product Flow

- Admin creates product -> DB insert -> invalidate product list cache.
- Authenticated user fetches product list/detail -> Redis first, DB fallback, then cache set.
- Update uses dynamic SQL for partial fields (`UpdateProductRequest` pointer fields).

### Cart Flow

- Add-to-cart resolves product price from `products` table.
- Insert uses UPSERT on `(user_id, product_id)` to increment quantity if item exists.
- Cart list endpoint is cached per user.

### Checkout Flow

1. Fetch all cart items for user.
2. Compute total by summing `item.Price * quantity`.
3. Insert order row.
4. Insert order_items rows for each cart line.
5. Delete cart rows for user.
6. Invalidate related cart/orders cache keys.
7. Return created order object.

## 11. Route and Middleware Topology

Router organization in `routes.GetRoutes`:

- Base router exposes `/` and `/healthz` publicly.
- Auth routes (`/auth/register`, `/auth/login`) remain public.
- `/api` subrouter applies `AuthMiddleware` to all nested routes.
- `adminRoute` is a subrouter of `/api` with additional `AdminMiddleware`.

This pattern centralizes auth requirements and keeps route declarations explicit.

## 12. Deployment and Local Runtime

### Docker Compose Services

- `go-app`: API container, code mounted at `/app`, publishes port `8000`.
- `go_db`: PostgreSQL 15, persistent volume `pgdata`.
- `go_redis`: Redis 7 alpine, persistent volume `redisdata`.
- `migrate`: one-shot migration runner (`migrate/migrate`) applying SQL `up` migrations.

### Dockerfile Behavior

- Base image: `golang:1.24-alpine3.21`.
- Installs `git`, `swag`, and `air`.
- Downloads dependencies (`go mod download`).
- Copies project source.
- Runs `swag init` during image build.
- Container command: `swag init && air` for auto-regeneration and live reload.

### Environment Variables Used

- `PORT`
- `JWT_SECRET`
- `DB_URL`
- `REDIS_URL`
- `REDIS_PASSWORD`
- `REDIS_DB`

## 13. Observed Project Characteristics

- API is intentionally simple and practical, with direct SQL in handlers/data layer.
- Caching is integrated consistently for read-heavy resources.
- Access control is role-driven and attached at routing level.
- Schema includes practical user profile extensions beyond minimal auth fields.
- Swagger annotations exist across handlers and generated docs are committed.

## 14. Migrations and Rollback

`up` migrations create:

1. `users`
2. `categories`
3. `products`
4. `cart_items`
5. `orders` and `order_items`

`down` migrations drop tables in reverse related order where needed (notably `order_items` before `orders`).

## 15. Typical Request Lifecycle Example

Example: `GET /api/products/42`

1. Client sends Bearer token.
2. `AuthMiddleware` validates token and injects `userID` context.
3. Handler checks Redis key `product_42`.
4. On miss: query PostgreSQL `products` by ID.
5. Cache the result for 5 minutes.
6. Return JSON product payload.

Example: `POST /api/orders/checkout`

1. Token validated and user resolved.
2. Cart rows fetched for that user.
3. Total computed server-side.
4. Order + order_items inserted.
5. User cart cleared.
6. Cache keys for user cart/orders invalidated.
7. Created order returned with `201`.

## 16. Documentation and Developer Experience

- Swagger UI is available at `/swagger/index.html` when server runs.
- `air` enables rapid local iteration inside container.
- SQL migrations are plain, readable, and versioned.
- Package layout is approachable for new contributors.

## 17. Summary

This project is a full backend foundation for an e-commerce system built with Go, PostgreSQL, and Redis. It includes authentication, role-based authorization, product/cart/order/category capabilities, migration-driven schema management, and generated API docs. The codebase is structured in a pragmatic layered style with explicit route security boundaries and cache-backed read paths that target practical performance for common commerce operations.
